## Context

仓库当前已经存在两套明显不同的前端气质：

- 认证页与安装页已经形成较成熟的视觉语言，体现在 `AuthShell`、`auth-panel-glass`、轻结构背景、低饱和品牌强调和更明确的空间节奏上。
- 已登录工作台仍以基础 token 和朴素壳层为主，`DashboardLayout` 只提供最基本的背景、侧边栏和内容 padding，缺少与登录页同源的层级表达。

与此同时，`DESIGN.md` 已经明确了系统设计方向不是营销化品牌页，也不是夸张科技风，而是“精致、克制、系统化”。这意味着统一主题改造的关键不在于复制登录页视觉，而在于提炼它背后的语言：更稳定的主题 token、更一致的壳层层级、更有识别度但不过度的品牌表达，以及可批量复用的页面表面模式。

这是一项跨页面、跨组件、跨模块的前端改造。proposal 阶段需要先把目标、原则、边界和推进方式写清，避免后续实现时演变为无边界的逐页重画。

## Goals / Non-Goals

**Goals:**
- 建立从登录页风格抽象出来的系统级主题模型，并落在全局 token 与布局壳层上。
- 明确统一主题改造目标：提升系统识别度与一致性，同时保留后台工作台的效率和信息密度。
- 明确设计原则：克制、系统化、同源但不照搬登录页舞台效果。
- 明确范围边界：优先做全局基础和高复用模式，不承诺一次性重做所有业务页面细节。
- 明确分阶段推进策略，使后续改造可以按阶段验收，而不是大爆炸切换。
- 提前记录风险、取舍与非目标，防止主题改造演变为品牌重塑项目。

**Non-Goals:**
- 不在本次 proposal 中实现任何页面或组件代码。
- 不把登录页的 orb、强背景结构线、玻璃舞台效果原样铺到全部业务页面。
- 不为每个业务模块单独定义一套品牌皮肤或模块专属主题。
- 不在本轮引入暗黑模式、多品牌换肤或运行时主题编辑器。
- 不修改后端接口、菜单模型或权限机制。

## Decisions

### D1: 以“认证页风格抽象”而不是“认证页复制”为统一主题方法

统一主题的来源是登录页已经验证过的视觉语言，但目标对象是已登录工作台，因此需要抽象出可复用原则，而不是复制具体特效。

保留的核心特征：
- 轻雾面、轻层级、低饱和品牌强调
- 更明确的前景/背景/容器层次
- 品牌色只做焦点和识别锚点

明确不直接迁移的元素：
- 满页舞台式背景
- 大尺寸装饰性 orb
- 强玻璃拟态容器

替代方案：直接把登录页样式扩散到全部页面。该方案实现快，但极易破坏后台场景的信息可读性，也与 `DESIGN.md` 的克制原则冲突，因此不采用。

### D2: 主题改造先落在 token 与壳层，再影响页面与组件

本次改造优先级分为三层：

1. 全局 token：补齐背景层、页面表面、导航表面、品牌强调、弱分隔和状态色的语义层次。
2. 布局壳层：统一 `DashboardLayout`、`TopNav`、`Sidebar` 的背景关系、边界关系和内容承载方式。
3. 页面与组件：在统一表面规则下逐步改造 page header、section、card、sheet、empty state 等高复用模式。

这样可以先解决“系统看起来像两套产品”的问题，再逐步替换模块细节，降低大范围同时改动带来的回归风险。

替代方案：直接按模块逐页翻新。该方案更容易出现局部好看但整体不统一的问题，也会重复制造 token 与容器规则，因此不采用。

### D3: 统一主题保持 `DESIGN.md` 的后台工作台基线

新主题必须兼容当前设计沉淀中的这些基线：

- 后台不是营销页
- 容器是轻表面，不是视觉主角
- 信息优先可读、可扫视
- 品牌识别点到为止

因此，统一主题的规范应约束业务页：
- 页面背景允许弱氛围，但主阅读区仍需稳定中性
- 卡片、分区、表单容器的层次差应主要靠表面和边框，而不是重阴影和重渐变
- 主色只能用于关键动作、激活态、状态锚点和少量品牌节点

替代方案：将登录页作为“更强品牌表达”的例外，业务页继续保持默认黑白灰。该方案会继续保留认证前后体验割裂，因此不采用。

### D4: 分阶段推进必须有清晰边界与验收口径

统一主题不应一次性覆盖全仓库全部页面。设计上按三阶段推进：

1. 基础阶段：主题 token、DashboardLayout、TopNav、Sidebar、通用 page shell。
2. 公共模式阶段：page header、section 容器、列表页、详情页、Sheet 表单、空状态、反馈组件。
3. 模块落地阶段：按业务模块批次替换旧视觉，优先高频页面和共享模式占比高的模块。

每阶段都应有完成标准，避免“主题改造永远进行中”。

替代方案：以页面数量为单位自由推进。该方案缺少验收基线，也难以同步 `DESIGN.md`，因此不采用。

### D5: Proposal 只定义边界，不预先承诺所有页面最终长相

在没有逐页探索和验证前，不应在 proposal 中把每个业务模块的最终视觉结构写死。规范层面只定义：

- 哪些壳层和共享模式必须统一
- 哪些视觉语言可以继承
- 哪些过度装饰手法禁止进入业务页

具体模块实现可在后续 apply 阶段结合现状局部调整，但不得偏离统一主题基线。

## Risks / Trade-offs

- **主题统一后业务页过度装饰** -> 通过规范明确禁止把认证页舞台效果整页迁移到工作台页面，并将“轻表面、弱氛围、强可读”写成硬约束。
- **token 调整影响范围过大** -> 先引入新增语义 token 或分层 token，再按阶段替换消费方，避免一次性重写所有现有组件。
- **模块页面风格不一致持续存在** -> 用“壳层先统一、模块后替换”的策略接受短期过渡，但要求共享模式先收敛，减少长期分裂。
- **设计改造影响开发节奏** -> 将任务拆成基础设施、共享模式、模块迁移三段，优先处理复用面最广的部分。
- **登录页风格被误解为紫色科技风模板** -> 设计原则中明确继承的是层级和气质，不是装饰元素；同时以 `DESIGN.md` 的“避免方向”作为约束。
- **DESIGN.md 与实现再次脱节** -> 将“稳定后更新 DESIGN.md”纳入任务收尾要求，而不是可选动作。

## Migration Plan

1. 在实现阶段先建立新的系统主题 token 与布局壳层，不直接大规模改业务页面。
2. 用共享页面模式替换最常见的旧容器和页面结构，建立可复用基线。
3. 分批迁移业务模块页面，并在每批完成后做视觉一致性回顾。
4. 当主题规则稳定且主要高频页面落地后，更新 `DESIGN.md` 记录新基线与淘汰模式。
5. 若某阶段出现可读性或回归问题，可暂时仅保留 token 与壳层改造，延后具体模块迁移。

## Open Questions

- 系统首页/仪表盘是否作为第一批模块落地点，还是先选择结构更稳定的管理类页面？
- 是否需要在主题 token 中预留“强调型页面”和“常规工作页”两档背景层次，以避免所有页面同质化？
- 现有共享组件中哪些已经足够稳定，可以直接升级；哪些需要先重命名或收口后再主题化？
- 主题改造完成后，是否需要补充一份专门的 UI review checklist，帮助后续页面继续遵循新基线？
