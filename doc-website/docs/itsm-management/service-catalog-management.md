# 服务目录

服务目录用于维护 ITSM 里可被提交、流转和交付的服务定义。你可以在这里按分类管理服务，补齐基础信息、SLA、协作规范、动作和流程，让一个服务从“可见”变成“可真正受理工单”。

## 一、这个模块解决什么问题

服务目录把“分类管理”“服务卡片总览”“服务详情配置”“动作管理”“流程设计”串在同一条链路里，适合用来搭建和持续维护 ITSM 服务。

当你需要新建一个服务、调整已有服务的处理方式，或在发布前确认服务是否能稳定运行时，优先进入这个页面。

## 二、谁会在什么场景下使用

- ITSM 管理员在新业务上线前创建服务、挂到合适分类，并配置 SLA 和运行方式。
- 流程负责人在已有服务上补充动作、设计经典流程，或为 Agentic 服务填写协作规范。
- 运维或平台负责人在发布前检查服务健康状态，确认是否存在缺动作、缺节点或引用失效等问题。

## 三、用户从哪个 Web 入口进入

- 左侧菜单：`ITSM > 服务目录`
- 默认入口：`/itsm/services`
- 详情入口：点击服务卡片进入 `/itsm/services/:id`
- 动作入口：服务详情页内进入“动作管理”，或直接进入 `/itsm/services/:id/actions`
- 流程入口：经典服务可进入 `/itsm/services/:id/workflow`

## 四、关键对象、状态与入口

- 第一眼会看到左侧分类导航和右侧服务卡片区。
- 左侧支持“全部”和分层分类视图，分类上会直接显示服务数量。
- 右侧每张服务卡片会展示服务名称、编码、所属分类、引擎类型和启用状态。
- 服务分为 `经典` 和 `智能` 两种引擎类型。
- 服务状态至少区分 `启用` 和 `停用`。
- 管理员可在列表页直接新增服务，也可在分类导航里新增、编辑、删除分类。
- 进入详情页后，基础信息、SLA 模板、协作规范、知识文档、发布健康检查都会汇总在同一页。

## 五、常见任务流程

### 任务 1：按分类创建一个新服务

路径：`ITSM > 服务目录 > 选择分类 > 新增服务`

步骤：
1. 在左侧分类树中选择目标分类，确认服务要归属在哪个目录。
2. 点击右上角“新增服务”。
3. 填写服务名称、服务编码、所属分类、引擎类型和描述。
4. 保存后系统会自动跳转到新服务详情页，继续补充 SLA、协作规范或流程内容。

结果：
- 新服务进入目录列表，可继续配置。

影响：
- 会新增一个可被引用和后续建单的服务定义。

可撤销性：
- 服务可删除，但删除前要先确认没有被业务依赖。

### 任务 2：把一个智能服务补齐到可发布状态

路径：`ITSM > 服务目录 > 服务详情`

步骤：
1. 打开目标服务详情页，确认服务基础信息、分类和 SLA 模板。
2. 在“协作规范”中补齐服务处理规则。
3. 点击“生成参考路径”，让系统基于协作规范生成策略草图。
4. 查看“发布健康检查”结果，逐条修复阻塞项、歧义项和失效引用。
5. 上传知识文档，补齐智能服务的知识上下文。

结果：
- 服务从仅有基础信息，推进到具备运行前检查依据的状态。

影响：
- 会直接影响后续智能工单能否稳定受理、决策和交付。

可撤销性：
- 协作规范、知识文档和生成结果都可以继续修改，但生成动作本身会刷新当前服务的运行建议。

### 任务 3：维护经典服务的流程和动作

路径：`ITSM > 服务目录 > 服务详情 > 工作流 / 动作管理`

步骤：
1. 进入经典服务详情页，确认当前服务为“经典”引擎。
2. 打开“工作流”页面，在流程设计器中调整节点、连线、进件表单和服务属性。
3. 如需执行自动化能力，再进入“动作管理”补充动作定义。
4. 保存流程后回到详情页，确认服务仍保持启用状态和正确的 SLA 绑定。

结果：
- 经典服务的审批链路、自动化动作和表单结构得到同步更新。

影响：
- 会直接改变后续工单的流转路径和自动化执行方式。

可撤销性：
- 后续可以继续改流程并再次保存，但每次保存都会影响新进入的工单处理路径。

## 六、风险动作与影响

- 删除服务是高风险动作，删除后该服务会从目录中移除，不适合在仍有业务依赖时直接执行。
- 删除分类前如果分类下仍有子分类或服务，页面会阻止删除；这说明目录层级本身也是业务约束。
- “生成参考路径”不是只读预览，它会刷新智能服务的运行建议和健康检查结果，适合在协作规范稳定后执行。
- 保存经典流程会改变后续工单实际执行链路，应在变更窗口内完成并复核节点、表单和动作引用。
- 停用服务不会删除定义，但会影响用户后续能否正常使用该服务。

## 七、与上下游模块的关系

- [工单监控](./ticket-monitoring.md)：服务目录决定工单从哪个服务发起，以及后续按什么流程流转。
- `服务台`：终端用户在服务台选择的可提交服务，来自这里维护的服务定义。
- `SLA 管理`：服务详情中的 SLA 模板来自 SLA 模块，决定响应和解决时效。
- `优先级管理`：服务流转后的工单优先级会影响监控和处理顺序。
- [引擎设置](./engine-settings.md)：智能服务依赖引擎设置中的模型、重试、超时和健康检查能力。
- `/ai/providers`：当详情页提示 Provider 未配置时，需要跳到 AI 模块先补基础模型能力。

## 八、常见误解、限制、边界条件

- “服务目录”不只是列表页，它同时承担服务设计入口，不能把它理解成纯粹的导航目录。
- 经典服务和智能服务共用同一目录，但详情页的可配置能力不同；经典服务重点是流程设计，智能服务重点是协作规范、知识和健康检查。
- 发布健康检查主要面向智能服务运行前风险，并不替代业务侧的最终验收。
- 分类数量和服务数量会直接显示在导航上，适合先按分类梳理再进详情，不建议在“全部”视图里硬找大规模服务。
