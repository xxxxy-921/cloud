# 引擎设置

引擎设置用于配置 ITSM 智能岗位背后的模型直连能力和运行保障规则。你可以在这里为服务受理岗、流程决策岗、SLA 保障岗以及参考路径生成、会话标题生成、发布健康检查等能力指定 Provider、模型、提示词、重试次数、超时时间和兜底处理边界。

## 一、这个模块解决什么问题

引擎设置把分散在智能工单链路里的核心运行参数集中到一个页面，避免每个服务各自维护一套模型和兜底规则。

当你需要让智能工单真正可运行、调整 AI 岗位的输出风格，或处理 Provider 缺失、模型未选、兜底人未配置等运行问题时，应进入这里。

## 二、谁会在什么场景下使用

- 平台管理员在初次启用 Agentic ITSM 前配置 Provider、模型和提示词。
- 运营或流程负责人在发现受理、决策或 SLA 处理效果不稳定时，回到这里调温度、重试和超时。
- 发布负责人在智能服务无法通过健康检查时，复核相关运行能力是不是已配置完整。

## 三、用户从哪个 Web 入口进入

- 左侧菜单：`ITSM > 引擎设置`
- 页面路由：`/itsm/engine-settings`
- 依赖跳转：当页面提示尚未配置 AI Provider 时，可直接跳到 `/ai/providers`

## 四、关键对象、状态与入口

- 页面分成两大区域：`模型直连能力` 和 `运行保障`。
- 模型直连能力覆盖 `服务受理岗`、`流程决策岗`、`SLA 保障岗`，以及 `参考路径生成`、`会话标题生成`、`发布健康检查`。
- 每个能力块都会显示当前状态：`就绪`、`需关注`、`异常`。
- 每块配置至少包含 Provider、模型、温度、最大重试、超时时间和系统提示词。
- 运行保障区域负责审计级别和兜底处理人，决定异常时记录多少，以及由谁接管。
- 系统提示词通过抽屉面板单独编辑，应用后即时生效。

## 五、常见任务流程

### 任务 1：为 ITSM 智能岗位完成基础建模

路径：`ITSM > 引擎设置`

步骤：
1. 先确认 AI 模块里已经存在可用 Provider 和模型。
2. 依次为服务受理岗、流程决策岗和 SLA 保障岗选择 Provider 与模型。
3. 调整温度、最大重试和超时时间。
4. 保存页面配置。

结果：
- 智能岗位具备最基本的运行能力，后续智能服务可复用这些设置。

影响：
- 会直接影响建单、决策和 SLA 风险处理的输出稳定性。

可撤销性：
- 配置可继续修改并重新保存，但变更会影响后续新产生的运行结果。

### 任务 2：编辑提示词并优化智能行为

路径：`ITSM > 引擎设置 > 某个能力块 > 编辑提示词`

步骤：
1. 打开目标能力块中的提示词编辑入口。
2. 在抽屉中修改系统提示词。
3. 点击“应用”，确认修改生效。
4. 如有必要，再保存整页配置并观察实际工单表现。

结果：
- 目标能力块的提示词即时更新。

影响：
- 会直接改变模型输出风格、摘要重点或健康检查判断口径。

可撤销性：
- 可以再次打开抽屉修改，但每次应用都会立即替换现行提示词。

### 任务 3：处理运行兜底和发布前缺口

路径：`ITSM > 引擎设置`

步骤：
1. 检查状态为“异常”或“需关注”的能力块。
2. 若缺 Provider 或模型，按提示跳到 AI 模块补齐。
3. 为异常兜底与审计区域设置合适的审计级别和兜底处理人。
4. 保存后回到智能服务详情页重新执行健康检查。

结果：
- 智能运行缺口被补齐，服务详情中的发布健康检查更容易通过。

影响：
- 会直接影响异常情况下是否有人接管，以及诊断信息是否足够。

可撤销性：
- 审计级别和兜底人都可以再调整，但调整后会改变后续异常处理与记录方式。

## 六、风险动作与影响

- 保存引擎设置会影响整个 ITSM 智能运行面，而不是单个服务。
- 修改系统提示词属于高影响操作，可能改变受理、决策和健康检查结论。
- 将最大重试或超时时间设得过高，会拉长故障恢复和用户等待时间；设得过低则可能导致智能岗位频繁中断。
- 关闭或弱化审计记录会降低排障能力，不适合在运行不稳定阶段使用。
- 不设置兜底处理人时，智能异常后的人工接管边界会变模糊。

## 七、与上下游模块的关系

- [服务目录](./service-catalog-management.md)：智能服务详情中的参考路径生成和发布健康检查依赖这里的模型能力。
- [工单监控](./ticket-monitoring.md)：工单里的 AI 决策、建议步骤和异常诊断，都会受到这里配置的影响。
- `/ai/providers`：Provider 缺失时必须先去 AI 模块补配置，ITSM 页面本身不会代替完成这一步。
- `智能岗位`：该页面与智能岗位配置同属系统配置入口，但前者更偏模型直连能力，后者更偏岗位分工。

## 八、常见误解、限制、边界条件

- “引擎设置”不是单服务级配置，而是 ITSM 领域级的共用底座。
- 页面里的“发布健康检查”能力不是手动审批工具，而是智能服务发布前的自动风险判断能力。
- 没有 Provider 时页面会给出跳转入口，这说明 AI 模块是前置依赖，不是可选增强项。
- 就算页面状态显示“就绪”，也只代表配置完整；实际业务效果仍需要回到服务详情和工单场景验证。
