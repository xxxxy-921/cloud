# DESIGN.md

Metis 前端设计沉淀文档。当前内容以【供应商管理】为准，记录已经落地并验证过的页面结构、视觉语言和交互规则。

## 设计定位

供应商管理不是营销页，也不是高装饰性的品牌展示区。

当前采用的设计方向是：

- 精致、克制、系统化
- 轻量卡片，不做厚重“业务卡”
- 信息密度适中，优先可读和可扫视
- 品牌识别只做点到为止，不抢整体系统主题
- 列表页和详情页属于同一套后台工作台语言

## 统一主题基线

### 安静视觉语言（当前基线）

全站工作区采用安静、轻量的视觉语言，以供应商详情页为参考基线。核心原则：**容器退到背景，内容成为主角。**

workspace-* CSS 类是唯一的视觉重量控制层，组件和页面通过继承自动获得统一风格：

- `workspace-surface`（Card 基底）：半透明背景 + 弱边框，无 blur、无渐变、无阴影
- `workspace-page-header`：纯结构布局（flex + padding），底线分隔（`border-b`），无容器装饰
- `workspace-table-card`（DataTableCard 基底）：同 workspace-surface，轻量容器
- `workspace-table-toolbar`（DataTableToolbar 基底）：纯结构，底线分隔
- `workspace-panel`（侧边栏）：轻背景 + 右边框，无 blur
- `workspace-shell-bg`（根背景）：纯色混合，无渐变层

视觉关键词：

- 半透明背景（opacity 0.4-0.6）
- 弱边框（opacity 0.5）
- 无 backdrop-blur
- 无 box-shadow
- 无 linear-gradient
- 间距和底线做结构分隔，不靠容器包裹
- 通用 `Card` 标题到内容的默认节奏使用紧凑间距（`gap-4`）；表单/详情配置卡片不额外叠加 `CardHeader pb-4`

### 不再使用

- 毛玻璃风格（backdrop-filter: blur）
- 白色渐变背景（linear-gradient white 74%→54%）
- 双层阴影（inset + drop shadow）
- color-mix 高白混合边框
- 厚重圆角容器包裹页面头部（rounded-[1.5rem] + 背景 + 阴影）
- 业务页继续维持默认黑白灰壳层
- `ghost` / `secondary` 按钮沿用与工作台表面脱节的旧灰阶样式
- 详情页和表单页使用裸标题行，缺少统一页面头层级

### 保留

- 认证页（auth-*）独立视觉体系不受影响
- 安装向导（install-*）独立视觉体系不受影响
- body 背景微妙渐变保留——容器透明后它提供纵深感
- 主按钮可以带品牌强调，普通操作优先使用轻表面按钮

## ITSM 智能工单设计

对应实现：`web/src/apps/itsm/components/smart-current-activity-card.tsx`、`web/src/apps/itsm/pages/tickets/[id]/index.tsx`

### 定位

智能工单详情页不是普通详情页，而是智能引擎的协同驾驶舱。

首屏必须优先回答三个问题：

1. 现在谁负责
2. AI 为什么这么做
3. 当前用户能做什么

### 结构规则

smart 工单首屏采用四段式：

```text
状态条：一句话说明当前发生什么
当前步骤主卡片：责任方 / 下一步 / 置信度 / SLA / 主操作
AI 驾驶舱：默认摘要，展开后看证据、表单字段、计划、工具/知识/动作信息
智能轨迹：展示运行中产生的 TicketActivity，不承担主要解释职责
```

时间线只作为审计记录，不再承担主要解释职责。

### 安全感规则

- AI 可以自动推进，但必须显式展示当前状态和下一步计划
- 低置信决策要前置确认入口，不把确认动作藏在时间线里
- 自动化动作要明确展示“执行中 / 已执行 / 失败”，失败时提供接管路径
- 管理员接管入口与普通处理动作并列呈现，但所有接管动作必须保留原因和审计记录
- smart 服务里的 `workflowJson` 在产品语义上是“参考路径/策略草图”，不是运行时严格执行的固定流程图

### 配置页规则

smart 服务配置页必须展示发布健康检查，至少覆盖：

- Agent 配置
- 协作规范
- 参考路径
- 知识库
- 自动化动作
- 兜底处理人
- 权限与接管

健康检查用于回答“这个服务是否足以安全运行”，不替代运行时审计。

## 认证页设计

对应实现：`web/src/pages/login/index.tsx`、`web/src/components/auth/auth-shell.tsx`

### 登录页定位

登录页不是营销落地页，也不是单纯的默认表单页。

当前采用的方向是：

- 控制台入口感，而不是“登录弹窗放大版”
- 品牌与系统语言并存，但品牌只做锚点，不做主视觉统治
- 桌面端采用双栏结构：左侧品牌叙事，右侧登录操作
- 移动端折叠成单列，优先保证输入与提交效率

避免的方向：

- 居中单卡 + 大面积发光 orb 的通用 SaaS 登录模板
- 过强玻璃拟态
- 为了显得先进而堆叠紫蓝科技特效
- 大段品牌介绍文案抢走登录动作焦点

### 布局规则

桌面端采用两栏：

1. 左栏：品牌识别、系统定位、一组轻量能力卡片
2. 右栏：登录表单、SSO、注册入口

结构如下：

```text
┌────────────────────────────────────────────────────────────┐
│ 品牌锚点 + 系统定位              登录标题 + 输入表单       │
│ 大标题 / 短说明                  用户名 / 密码 / 验证码    │
│ 3 个轻量能力卡                   登录 / SSO / 注册入口     │
│ 底部状态标签 / 版本信息                                   │
└────────────────────────────────────────────────────────────┘
```

要求：

- 左栏负责气质建立，不承担复杂交互
- 右栏必须保持明确的主任务路径
- 登录卡与外层舞台容器要有层级差，但不能做厚重卡片

### 视觉语言

- 背景允许有极弱结构线和轻量色团，但只能做氛围，不可喧宾夺主
- 外层容器是“薄雾面工作台”，不是高亮玻璃卡
- 登录卡使用细描边、低阴影、浅雾面背景
- 主色只用于焦点、按钮和品牌符号，不做大面积铺色

### 品牌表达

品牌表达保留在以下位置：

1. `AuthBrandLockup`
2. 左栏标题与短说明
3. 少量主色强调

不再使用：

- 满屏品牌色渐变
- 把品牌口号做成大段说明区
- 用背景特效替代品牌识别

### 表单规则

- 标题应直接说明动作，例如 `登录 Metis`
- 副标题只说明进入什么系统，不重复“请输入账号继续”这类空话
- 输入框使用主题 token，不单独维护 auth 专属灰阶体系
- SSO 按钮是次级动作，样式应回到系统按钮语言
- 错误提示必须内联、清晰、可即时感知

### 文案基线

- 优先使用明确动作：`登录`、`注册`、`验证`
- 少用模糊 CTA，如 `继续`
- 左栏说明控制在一段内，最多一句定位 + 一句补充
- 标签文案偏系统能力，不写营销化 slogan

## 供应商管理信息架构

采用两层结构：

1. 列表页：供应商网格卡片
2. 详情页：单个供应商的模型工作区

适用原因：

- 供应商本身需要快速总览
- 供应商下面还挂着模型管理，适合进入详情页做完整操作
- 列表页负责识别、状态、快速测试、进入详情
- 详情页负责模型的增删改、默认模型设置、同步

## 列表页设计

对应实现：`web/src/apps/ai/components/provider-card.tsx`

### 网格

供应商列表使用中等密度卡片网格。

原则：

- 卡片不能过小，否则品牌、状态、模型类型都会挤压
- 也不能过大，否则会喧宾夺主，破坏系统页面节奏
- 当前卡片高度为略高于菜单卡片的一档，用于容纳模型类型 chips 和底部状态区

### 卡片结构

当前卡片分为三层：

1. 头部
2. 模型类型摘要
3. 底部状态与快捷操作

结构如下：

```text
┌──────────────────────────────────────┐
│ [Logo] 名称  类型                ⋯  │
│        baseUrl                        │
│                                [chips]│
│ ──────────────────────────────────── │
│ ● 状态 · 相对时间          连通测试   │
└──────────────────────────────────────┘
```

### 头部规则

- 左侧使用真实供应商 logo，不再使用字母占位
- logo 放在中性容器里：`rounded-xl + border + muted 背景`
- 标题使用 `text-base font-semibold`
- 类型使用小号 `outline badge`
- `baseUrl` 是唯一保留的副信息
- 不显示 API Key

### 品牌表达

品牌表达只保留在两处：

1. 真实 logo
2. 类型 badge 文案

不再使用：

- 顶部粗色条
- 大面积品牌底色
- 品牌主导的卡片视觉

原因：

- 供应商是后台管理对象，不是品牌展示对象
- 真实 logo 已足够提供识别，不需要再用颜色重复强调

### 模型摘要规则

列表页中部不显示总模型数文案，例如：

- 不推荐：`模型数 14`

当前改为展示“各模型类型数量”，例如：

- `大语言模型 12`
- `向量模型 1`

这些信息使用轻量 `outline badge`：

- 小尺寸
- 中性色
- 不抢标题
- 右对齐排列

如果未来模型类型过多：

- 优先裁剪展示顺序靠前的类型
- 不建议把 chips 扩成第二层大模块

### 底部规则

底部使用 `border-t` 与上方内容分隔。

左侧：

- 状态圆点
- 状态文案
- 相对时间（如 `37m`）

右侧：

- 高频操作只保留一个：`连通测试`

中低频操作：

- 编辑、删除进入右上角 `⋯`

### hover 与操作区

- hover 只做轻微边框和阴影变化
- 不使用明显位移或厚阴影
- 卡片点击进入详情页
- 所有按钮和菜单必须挂 `data-action-zone`，避免点击穿透

### 引导卡

新增供应商使用虚线引导卡，样式比普通卡片更弱：

- `border-dashed`
- 中性背景
- `+` 图标 + 文案

引导卡必须与普通卡片使用同一尺寸体系，否则列表会断层。

## 智能体能力绑定

助手/编码智能体的能力绑定采用“集合优先”的交互模型：

- 外层卡片表示能力集合（Set），例如工具集合、MCP 集合、技能包集合、知识库集合、知识图谱集合
- 卡片展示集合名称、描述、已选数量/总数
- 点击集合卡片打开 Sheet，在 Sheet 中勾选或取消勾选该集合内的具体能力项
- 运行时只暴露已绑定集合中被显式勾选的可用子项；新增到集合中的子项不会自动授权给已有智能体
- 详情页按集合展示绑定摘要，不再只按资源类型展示 flat item 数量

视觉上延续工作台轻表面规则：集合卡片使用弱边框、轻背景、紧凑标题节奏，避免把能力项直接全部平铺在页面上。

## 详情页设计

对应实现：`web/src/apps/ai/pages/providers/[id].tsx`

### 头部信息区

详情页顶部不使用“概览卡”。

当前采用平铺信息区：

- 左侧：logo + 名称 + 类型
- 同一视觉层里展示 `baseUrl / 状态 / 协议 / 最后检测`
- 右侧：操作按钮组

原则：

- 不做单独的统计卡
- 不做大面积信息容器
- 头部只承担身份识别和全局操作

头部禁止：

- 返回按钮占一整块区域
- 重复页面标题
- 把状态、协议、最后检测拆成稀疏 description-list

这些信息应该压缩在同一行或同一信息层里，避免浪费垂直空间。

### 头部动作

当前保留三个操作：

- `连通测试`
- `同步模型`
- `编辑`

样式统一：

- `outline` 小按钮
- 同一高度
- 语义平级，不做主次冲突

当模块已经接入统一主题后，详情页头部外层优先复用 `workspace-page-header`：

- 返回路径放在头部信息区上方，作为轻量 breadcrumb/返回入口
- 标题、状态、类型标签放在同一信息块内
- 右侧动作区继续使用小尺寸按钮，但视觉要和工作台按钮语言一致

## 模型工作区设计

### 整体结构

详情页下半区不再用一个“模型管理”大面板包住所有内容。

当前结构是：

- 直接按模型类型拆分为多个分区面板
- 每种模型类型一个面板
- 桌面端一行 2 个面板

原因：

- 每种类型是天然分组
- 比单个大表格更适合扫视和定位
- 比 tabs 更适合比较不同类型是否有数据

### 分区面板风格

模型类型面板不是“强卡片”，而是“轻分区”。

规则：

- 弱边框
- 轻背景
- 大圆角，但不过度强调卡片感
- 标题区和内容区只用一条轻分隔线区分

不使用：

- 厚重标题条
- 大色块表头
- 数量统计 badge
- 很强的容器投影

### 分区头部

每个面板头部只保留三类元素：

1. 类型标题
2. 面板内搜索
3. 新建模型按钮

#### 搜索

搜索必须在各个类型面板内部，不放全局统一搜索。

原因：

- 用户搜索时往往针对某个类型
- 全局搜索会抢占页面结构中心
- 面板内搜索更符合“局部工作区”语义

当前搜索样式要求：

- 扁平
- 透明背景
- 无胶囊感
- 无明显边框盒子
- 仅保留轻底线

不允许使用高凸起、圆润胶囊、厚阴影输入框。

#### 新建模型

“新建模型”放在对应类型面板内，而不是全局工具栏。

点击后：

- 直接创建该类型模型
- `ModelSheet` 预填 `defaultType`

这是更短的操作路径，也减少了用户在表单里重复选择类型。

### 面板内容

每个面板内部仍然用表格展示模型，但表格只是承载数据，不应成为视觉主角。

当前列包含：

- 显示名称
- 模型 ID
- 状态
- 默认
- 操作

原则：

- 行高适中
- 默认列使用星标图标，不写冗余文案
- 操作列图标化
- 操作列标题和内容居中对齐

### 操作列

当前操作列统一使用图标按钮：

- 设为默认：`Star`
- 编辑：`Pencil`
- 删除：`Trash2`

原因：

- 文本按钮在窄面板里过于拥挤
- 图标化后更适合双列面板布局

规范：

- 操作图标水平居中排列
- 删除图标使用 `destructive`
- 破坏性操作必须二次确认

如果后续可用性不足，可补 tooltip，但不退回文字按钮。

### 空状态

每个类型面板都允许为空。

空状态设计：

- 不收起面板
- 保持固定高度
- 中间展示轻量空态

原因：

- 保持整体网格稳定
- 用户能明确知道该类型“目前没有模型”，而不是“不存在这个类型区域”

### 分页

每个类型面板独立分页：

- 每页 5 条
- 面板固定高度
- 底部有分页栏

分页栏规则：

- 可点击页码
- 保留上一页 / 下一页按钮
- 页数较多时用省略号

不推荐只显示 `3 / 3` 这种不可点击信息，因为跳页效率太低。

## Logo 规则

供应商 logo 当前使用 LobeHub 的静态 SVG 资源：

- 依赖：`@lobehub/icons-static-svg`
- 本地封装：`web/src/apps/ai/components/provider-logo.tsx`

选择静态 SVG 而不是重 React 组件包的原因：

- 依赖更轻
- 不引入额外 UI 体系
- 足够满足品牌识别

优先支持：

- OpenAI
- Anthropic
- Ollama

未知类型 fallback 才允许退回到文本缩写。

## 当前供应商管理设计关键词

如果后续继续扩展这个模块，必须保持以下关键词一致：

- 中性系统感
- 精致而克制
- 品牌识别点到为止
- 信息聚合但不做大统计
- 局部工作区优先于全局工具栏
- 表格可用，但不要让表格统治页面
- 卡片是轻容器，不是视觉主角

## 不再使用的旧模式

以下模式已经被当前供应商管理 UI 淘汰，不应继续沿用：

- 顶部品牌色条作为主视觉
- Avatar 首字母模拟品牌识别
- 详情页顶部“返回 + 标题 + 大概览卡”结构
- 全局统一模型搜索框
- 单个大面板包裹所有模型管理内容
- `模型数 14` 这种总数型摘要
- 强卡片感的模型类型面板
